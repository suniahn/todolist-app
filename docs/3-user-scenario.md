## 개요

본 문서는 TodoListApp의 주요 사용 흐름을 **사용자 행동 중심**으로 설명하는 시나리오 모음입니다. 각 시나리오는 특정 페르소나의 목표 달성 과정에서 발생하는 단계별 상호작용, 시스템 응답, 예외 상황, 사후 상태 변화를 포함합니다. 개발팀이 UI/UX 설계 및 테스트 케이스 도출 시 참고하며, 제품팀이 기능 우선순위를 검증하는 기준 문서입니다.

### SCN-01. 신규 사용자 회원가입 및 첫 할일 등록

**페르소나:** 김지수 (26세, 마케팅팀 신입)  
**관련 UC:** UC-01 (회원가입), UC-04 (할일 등록)  
**관련 BR:** BR-01 (미인증 차단), BR-07 (카테고리 필수), BR-08 (날짜 유효성)

#### 사전 조건
- 사용자가 아직 가입하지 않은 상태
- 브라우저에서 `/register` 페이지 접근 가능

#### 시나리오 흐름

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 1 | 회원가입 페이지에서 이메일 입력 (예: `jisukim@company.com`) | 유효한 이메일 형식 확인, 중복 검사 시작 |
| 2 | 비밀번호 입력 (`Password123`) | 클라이언트: 최소 8자, 영문+숫자 검증 통과 표시 |
| 3 | 이름 입력 (`김지수`) | 유효성 검사 통과 (1~50자) |
| 4 | "회원가입" 버튼 클릭 | POST `/api/v1/auth/register` 호출 |
| 5 | — | 서버: 이메일 중복 확인, 비밀번호 bcrypt 해시, users 테이블 INSERT |
| 6 | — | 도메인 이벤트 `UserRegistered` 발생 → 기본 카테고리 4종 (업무, 개인, 건강, 쇼핑) 접근 허용 |
| 7 | — | HTTP 201 반환, 자동으로 `/` (할일 목록) 페이지로 리다이렉트 |
| 8 | 화면에서 "새로운 할일 추가" 버튼 클릭 | `/todos/new` 페이지 로드 |
| 9 | 제목 입력 (`[마케팅] 상반기 전략 보고서 작성`) | 실시간 글자 수 표시 (현재 0/200) |
| 10 | 설명 입력 (선택) (`Q1 마케팅 성과 분석 및 개선안 포함`) | 입력값 표시 |
| 11 | 시작일 선택 (`2026-05-13`) | 달력 또는 date input에서 선택 |
| 12 | 종료예정일 선택 (`2026-05-20`) | 시작일 이후의 날짜 선택 |
| 13 | 카테고리 드롭다운에서 "업무" 선택 | 4개 기본 카테고리 노출 |
| 14 | "등록" 버튼 클릭 | POST `/api/v1/todos` 호출 (user_id, category_id, 제목, 설명, 날짜) |
| 15 | — | 서버: 종료예정일 ≥ 시작일 검증 통과, `is_completed = false`, `completed_at = NULL`로 INSERT |
| 16 | — | 도메인 이벤트 `TodoCreated` 발생 |
| 17 | — | HTTP 201 반환, 할일 목록 페이지로 리다이렉트 |
| 18 | 목록에서 방금 등록한 할일 확인 | 제목, 카테고리(업무), 시작일~종료예정일, 미완료 상태 표시 |

#### 예외 흐름

**E1. 이메일 중복 시**
- 단계 5 대신: 서버가 이메일 중복 감지
- HTTP 409 반환, 에러 코드: `AUTH_EMAIL_DUPLICATE`
- 사용자 메시지: "이미 사용 중인 이메일입니다"
- UI: 이메일 입력 필드 하단에 빨간색 오류 메시지 표시
- 사용자: 다른 이메일로 재시도

**E2. 비밀번호 조건 미충족 시**
- 단계 2 대신: 클라이언트가 실시간 검증
- "비밀번호는 최소 8자 이상이며 영문과 숫자를 포함해야 합니다" 메시지
- "회원가입" 버튼 비활성화
- 사용자: 조건에 맞게 수정

**E3. 할일 등록 시 종료예정일 < 시작일 시**
- 단계 15 대신: 서버 유효성 검사 실패
- HTTP 400 반환, 에러 코드: `TODO_INVALID_DATE`
- 사용자 메시지: "종료예정일은 시작일보다 이전일 수 없습니다"
- UI: 종료예정일 필드 하단에 오류 표시
- 사용자: 날짜 수정 후 재시도

#### 사후 조건
- users 테이블에 새 레코드 생성 (id, email, hashed_password, name, created_at, updated_at)
- categories 테이블에서 `is_default=true, user_id=NULL`인 4개 레코드를 사용자가 접근 가능
- todos 테이블에 새 레코드 생성 (user_id, category_id='업무', title, description, start_date, due_date, is_completed=false, completed_at=NULL)
- 클라이언트: Access Token · Refresh Token 모두 Zustand authStore(메모리)에 저장 (페이지 새로고침 시 초기화)
- 화면: 할일 목록 페이지에 방금 등록한 할일이 종료예정일 오름차순으로 표시

### SCN-02. 로그인 실패 후 재시도 및 성공

**페르소나:** 최미경 (48세, 총무팀 과장)  
**관련 UC:** UC-02 (로그인)  
**관련 BR:** BR-02 (이메일 유일성)

#### 사전 조건
- 사용자 미경은 이미 가입한 상태 (이메일: `mikyung.choi@company.com`, 비밀번호: `SecurePass123`)
- 브라우저에서 `/login` 페이지 접근

#### 시나리오 흐름

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 1 | 로그인 페이지에서 이메일 입력 (`mikyung.choi@company.com`) | 입력값 표시 |
| 2 | 비밀번호 입력 (잘못된 값: `WrongPassword`) | 입력값 마스킹 처리 |
| 3 | "로그인" 버튼 클릭 | POST `/api/v1/auth/login` 호출 |
| 4 | — | 서버: 이메일 존재 확인 → 비밀번호 bcrypt 비교 실패 |
| 5 | — | HTTP 401 반환, 에러 코드: `AUTH_INVALID_CREDENTIALS` |
| 6 | — | 사용자 메시지: "이메일 또는 비밀번호가 올바르지 않습니다" (단일 메시지, 이메일/비밀번호 구분 안 함) |
| 7 | UI: 로그인 폼의 상단에 빨간 토스트 메시지 표시, 입력 필드는 유지 | — |
| 8 | 사용자가 비밀번호 필드를 선택하여 수정 (`SecurePass123` 입력) | 입력값 마스킹 처리 |
| 9 | "로그인" 버튼 클릭 | POST `/api/v1/auth/login` 호출 |
| 10 | — | 서버: 이메일 존재 확인 → 비밀번호 bcrypt 비교 성공 |
| 11 | — | JWT Access Token (1시간 만료) + Refresh Token (7일 만료) 생성 |
| 12 | — | HTTP 200 반환, 응답 본문에 Access Token · Refresh Token 모두 포함 |
| 13 | — | 도메인 이벤트 `UserLoggedIn` 발생 (감사 로그 등 향후 사용) |
| 14 | 클라이언트: Access Token · Refresh Token을 Zustand authStore(메모리)에 저장 | — |
| 15 | 자동으로 `/` (할일 목록) 페이지로 리다이렉트 | 페이지 로드 시 Access Token 헤더 포함 |
| 16 | 할일 목록 조회 (사용자의 할일만 표시) | GET `/api/v1/todos` 응답 (200건 limit 내에서) |

#### 예외 흐름

**E1. 등록되지 않은 이메일 입력 시**
- 단계 4 대신: 서버가 이메일을 찾지 못함
- HTTP 401 반환, 에러 코드: `AUTH_INVALID_CREDENTIALS`
- 동일한 사용자 메시지: "이메일 또는 비밀번호가 올바르지 않습니다"

**E2. 네트워크 오류 (요청 타임아웃) 시**
- 단계 9 대신: 클라이언트가 네트워크 오류 감지
- UI: "네트워크 오류가 발생했습니다. 잠시 후 다시 시도해주세요" 메시지
- "다시 시도" 버튼 제공

#### 사후 조건
- 클라이언트: Access Token · Refresh Token 모두 Zustand authStore(메모리)에 저장
- 서버: 감사 로그에 로그인 시도 기록 (향후 보안 모니터링용)
- 화면: 할일 목록 페이지에 최미경의 할일들만 표시, 상단에 "최미경님 환영합니다" 메시지 (선택사항)

### SCN-03. 할일 등록 시 날짜 유효성 오류 처리

**페르소나:** 김지수 (26세, 마케팅팀 신입)  
**관련 UC:** UC-04 (할일 등록)  
**관련 BR:** BR-08 (종료예정일 ≥ 시작일)

#### 사전 조건
- 사용자가 로그인 상태
- `/todos/new` 할일 등록 페이지 접근

#### 시나리오 흐름

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 1 | 제목 입력 (`SNS 마케팅 콘텐츠 작성`) | 입력값 표시 |
| 2 | 설명 입력 (선택) | 입력값 표시 |
| 3 | 시작일 선택 (달력에서 `2026-05-20` 선택) | 날짜 필드에 "2026-05-20" 표시 |
| 4 | 종료예정일 선택 시도 (실수로 `2026-05-15` 선택) | 날짜 필드에 "2026-05-15" 표시 |
| 5 | 클라이언트 실시간 검증 | "종료예정일은 시작일 이후여야 합니다" 경고 메시지 표시 (노란색) |
| 6 | UI: 종료예정일 필드에 빨간 테두리 표시 | — |
| 7 | "등록" 버튼 시도 클릭 | 버튼이 비활성화되어 있음 (유효성 검사 미통과) |
| 8 | 사용자가 종료예정일 수정 (`2026-05-25` 선택) | 날짜 필드 갱신 |
| 9 | 클라이언트 실시간 검증 | 경고 메시지 사라짐, "종료예정일 필드 빨간 테두리" 제거 |
| 10 | "등록" 버튼 활성화 | — |
| 11 | 카테고리 선택 (`개인`) | 드롭다운에서 선택 |
| 12 | "등록" 버튼 클릭 | POST `/api/v1/todos` 호출 (start_date=2026-05-20, due_date=2026-05-25) |
| 13 | — | 서버: 유효성 검사 확인 (due_date ≥ start_date) 통과 |
| 14 | — | HTTP 201 반환, 새 todo 레코드 생성 |
| 15 | — | 목록 페이지로 리다이렉트 |

#### 예외 흐름

**E1. 클라이언트 검증을 우회한 경우 (예: API 직접 호출)**
- 단계 12 대신: POST 요청에서 due_date < start_date인 경우
- 서버: DB 제약조건 `CHECK (due_date >= start_date)` 또는 비즈니스 로직에서 검증
- HTTP 400 반환, 에러 코드: `TODO_INVALID_DATE`
- 사용자 메시지: "종료예정일은 시작일보다 이전일 수 없습니다"

**E2. 시작일과 종료예정일이 같은 경우**
- 허용됨 (due_date = start_date는 비즈니스 규칙 상 가능)
- 서버: due_date ≥ start_date 조건 통과
- 클라이언트: 경고 메시지 미표시

#### 사후 조건
- todos 테이블에 새 할일 INSERT (start_date=2026-05-20, due_date=2026-05-25)
- 할일 목록에서 종료예정일 오름차순으로 정렬되어 표시
- 이 할일은 2026-05-25까지 `schedule_status: ongoing`으로 조회됨

### SCN-04. 카테고리·기간초과 필터로 밀린 할일 파악

**페르소나:** 박성준 (35세, 개발팀 파트장)  
**관련 UC:** UC-08 (할일 목록 조회·필터)  
**관련 BR:** BR-03 (본인 할일만 조회)

#### 사전 조건
- 박성준이 로그인 상태
- 이미 여러 할일이 등록되어 있음:
  - 할일 1: 카테고리='업무', 종료예정일='2026-05-10' (이미 지남), 미완료
  - 할일 2: 카테고리='업무', 종료예정일='2026-05-18', 미완료
  - 할일 3: 카테고리='개인', 종료예정일='2026-05-12' (이미 지남), 완료됨
  - 할일 4: 카테고리='업무', 종료예정일='2026-05-30', 완료됨
- `/` (할일 목록) 페이지 접근

#### 시나리오 흐름

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 1 | 페이지 로드 | 기본 조회: 모든 미완료 할일을 종료예정일 오름차순으로 표시 |
| 2 | — | 화면: 할일 1, 2 보임 (완료된 할일 4는 제외, 완료된 할일 3은 이미 지났지만 completed_at 있으므로 제외) |

| 1 | 개발자 도구 또는 URL 직접 입력: `/todos/todo-999` | 요청 시도 |
| 2 | — | GET `/api/v1/todos/todo-999` 호출 (Authorization 헤더: Access Token 포함) |
| 3 | — | 서버: JWT 토큰 검증 → user_id=김지수 추출 |
| 4 | — | 서버: 해당 todo의 user_id 확인 → user_id=박성준 (다름) |
| 5 | — | HTTP 403 Forbidden 반환, 에러 코드: `TODO_FORBIDDEN` |
| 6 | — | 사용자 메시지: "접근 권한이 없습니다" |
| 7 | UI: 에러 페이지 또는 403 메시지 표시 | — |

**Case 2: 타인 할일 수정 시도**

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 1 | 개발자 도구에서 PUT `/api/v1/todos/todo-999` 직접 호출 | — |
| 2 | — | 요청 본문: {title='변조된 제목', ...} |
| 3 | — | 서버: 토큰 검증 → user_id=김지수 |
| 4 | — | 서버: 해당 todo 조회 → user_id=박성준 (다름) |
| 5 | — | 수정 진행 안 함 |
| 6 | — | HTTP 403 Forbidden 반환, 에러 코드: `TODO_FORBIDDEN` |
| 7 | — | 사용자 메시지: "접근 권한이 없습니다" |

**Case 3: 타인 할일 삭제 시도**

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 1 | 개발자 도구에서 DELETE `/api/v1/todos/todo-999` 호출 | — |
| 2 | — | 서버: 토큰 검증 → user_id=김지수 |
| 3 | — | 서버: 해당 todo 조회 → user_id=박성준 (다름) |
| 4 | — | HTTP 403 Forbidden 반환, 에러 코드: `TODO_FORBIDDEN` |

#### 예외 흐름

**E1. Access Token이 없거나 유효하지 않은 경우**
- 단계 3 대신: 토큰 검증 실패
- HTTP 401 Unauthorized 반환, 에러 코드: `AUTH_UNAUTHORIZED`
- 사용자 메시지: "로그인이 필요합니다"
- UI: 로그인 페이지로 리다이렉트

**E2. 존재하지 않는 할일 조회 시도 (타인 소유 여부와 무관)**
- 단계 4 대신: 해당 ID의 레코드가 없음
- 동일하게 HTTP 403 또는 HTTP 404 반환
- 보안 상 403이 권장됨 (존재 여부 노출 방지)

#### 사후 조건
- 타인의 할일에 어떠한 변조도 발생하지 않음
- 요청자 (김지수)는 "접근 권한이 없습니다" 에러만 수신
- 서버 로그: 권한 검증 실패 기록 (감사 목적)
- 박성준의 할일 `todo-999`는 그대로 유지됨

### SCN-09. Access Token 만료 후 자동 재발급 흐름

**페르소나:** 공통 (인증 흐름)  
**관련 UC:** UC-02 (로그인), 토큰 재발급  
**관련 BR:** BR-01 (미인증 차단)

#### 사전 조건
- 사용자가 로그인한 지 59분이 경과 (Access Token 만료 약 1분 전)
- 할일 목록 페이지에서 작업 중
- Refresh Token은 Zustand authStore(메모리)에 유효 (7일 유효)

#### 시나리오 흐름

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 1 | 할일 목록에서 새로운 할일 등록 버튼 클릭 | POST `/api/v1/todos` 호출 (Authorization: Bearer <Access Token>) |
| 2 | — | 서버: Access Token 검증 |
| 3 | — | JWT exp 확인 → 만료됨 (iat + 3600 < 현재시간) |
| 4 | — | HTTP 401 Unauthorized 반환, 에러 코드: `AUTH_TOKEN_EXPIRED` |
| 5 | 클라이언트: (자동 인터셉터) 401 응답 감지 | TanStack Query 또는 Axios 인터셉터 작동 |
| 6 | — | POST `/api/v1/auth/refresh` 호출 (Refresh Token은 authStore에서 Authorization 헤더로 전달) |
| 7 | — | 서버: Refresh Token 검증 |
| 8 | — | Refresh Token 유효 (exp: 7일) → 새로운 Access Token 발급 |
| 9 | — | 새 Access Token 응답 (응답 본문 또는 Authorization 헤더) |
| 10 | 클라이언트: Zustand authStore의 Access Token 갱신 | — |
| 11 | — | 원래 요청 재시도: POST `/api/v1/todos` (새 Access Token 포함) |
| 12 | — | 서버: 새 Access Token 검증 성공 |
| 13 | — | 할일 등록 진행 |
| 14 | — | HTTP 201 반환, 새 todo 객체 응답 |
| 15 | 사용자는 아무 인지 없이 할일이 정상 등록됨 | 목록에 새 할일 표시 |

#### 예외 흐름

**E1. Refresh Token도 만료된 경우 (7일 초과)**
- 단계 8 대신: Refresh Token 검증 실패
- HTTP 401 Unauthorized 반환, 에러 코드: `AUTH_TOKEN_EXPIRED` 또는 `AUTH_UNAUTHORIZED`
- 클라이언트: Zustand authStore 초기화 (Access Token · Refresh Token 삭제)
- 사용자 메시지: "인증이 만료되었습니다. 다시 로그인해주세요"
- UI: 로그인 페이지로 리다이렉트

**E2. 자동 재발급 중 네트워크 오류 시**
- 단계 6 대신: `/auth/refresh` 요청 실패
- 클라이언트: 재시도 로직 (최대 3회)
- 3회 실패 후: 로그인 페이지로 리다이렉트

**E3. Refresh Token이 없는 경우 (authStore 비어있음)**
- 단계 6 대신: Refresh Token 없이 요청
- HTTP 401 반환
- 사용자 메시지: "로그인이 필요합니다"
- UI: 로그인 페이지로 리다이렉트

#### 사후 조건
- Zustand authStore Access Token 갱신: 새로운 JWT (exp: now + 3600초)
- Zustand authStore Refresh Token 유지: 기존 값 (exp: now + 7일)
- 원래 요청 완료: 할일 등록 성공
- 사용자 경험: 명시적인 재로그인 없이 투명하게 처리됨

### SCN-10. 로그아웃 후 보호 경로 접근 차단

**페르소나:** 공통 (보안 검증)  
**관련 UC:** 로그아웃  
**관련 BR:** BR-01 (미인증 사용자는 할일 기능 접근 불가)

#### 사전 조건
- 사용자가 로그인 상태, 할일 목록 페이지에 있음
- Zustand authStore에 Access Token · Refresh Token 저장됨

#### 시나리오 흐름

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 1 | 우상단 "로그아웃" 버튼 클릭 | — |
| 2 | 확인 다이얼로그 (선택사항): "정말 로그아웃하시겠습니까?" | — |
| 3 | "로그아웃" 확인 | POST `/api/v1/auth/logout` 호출 |
| 4 | — | 서버: 현재 사용자의 Refresh Token 무효화 (선택: DB에 블랙리스트 추가 또는 단순 상태 변경) |
| 5 | — | HTTP 200 또는 204 반환 |
| 6 | 클라이언트: Zustand authStore 초기화 (Access Token · Refresh Token · 사용자 정보 삭제) | — |
| 7 | — | 서버: Refresh Token 무효화 처리 완료 |
| 8 | TanStack Query 캐시 전체 초기화 (할일 목록 등 서버 상태 제거) | — |
| 9 | 자동으로 `/login` 페이지로 리다이렉트 | — |

#### 보호 경로 접근 차단

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 10 | 로그아웃 후 브라우저 주소창에서 `/` 직접 입력 | 클라이언트: 라우터에서 인증 상태 확인 |
| 11 | — | Access Token이 없음 (Zustand authStore 비어있음) |
| 12 | — | 클라이언트: `/login` 페이지로 리다이렉트 (또는 `/register`) |
| 13 | 사용자는 로그인 화면만 볼 수 있음 | — |

#### 서버 레벨 검증

| 단계 | 사용자 행동 (직접 API 호출) | 시스템 응답 |
|------|------------|------------|
| 14 | 개발자 도구에서 수동으로 GET `/api/v1/todos` 호출 (Access Token 미포함) | — |
| 15 | — | 서버: Authorization 헤더 없음 |
| 16 | — | HTTP 401 Unauthorized, 에러 코드: `AUTH_UNAUTHORIZED` |
| 17 | — | 사용자 메시지: "로그인이 필요합니다" |

#### 예외 흐름

**E1. 로그아웃 요청 중 네트워크 오류 시**
- 단계 3 대신: POST `/api/v1/auth/logout` 실패
- 클라이언트: Zustand authStore 즉시 초기화 (낙관적 업데이트)
- `/login` 페이지로 리다이렉트
- 서버: Refresh Token 무효화 안 됨 (최대 7일간 재사용 가능하나, 클라이언트 메모리가 초기화되어 Refresh Token 자체가 없으므로 재사용 불가)

**E2. 로그아웃 후 Back 버튼으로 이전 페이지 접근 시도**
- 브라우저 히스토리에 이전 페이지(/todos) 존재
- Back 버튼 클릭 → 페이지 로드 시도
- 클라이언트: Access Token 없음 감지 → 로그인 페이지로 리다이렉트
- 또는 서버: 요청에 Access Token 없음 → 401 반환 → 클라이언트 리다이렉트

#### 사후 조건
- Zustand authStore: Access Token · Refresh Token · 사용자 정보 모두 삭제됨 (currentUser = null)
- TanStack Query 캐시: 초기화됨 (todos = [])
- 서버: (선택) Refresh Token 블랙리스트 또는 무효 상태로 마크
- 사용자는 모든 보호 경로 (`/`, `/todos/*`, `/categories` 등) 접근 불가
- 로그인 페이지(`/login`) 및 회원가입 페이지(`/register`)만 접근 가능

## 요약: 시나리오 기반 개발 체크리스트

개발팀이 각 시나리오를 구현 및 테스트할 때 참고할 항목:

### API 엔드포인트 검증
- [ ] POST `/auth/register` — 이메일 중복, 유효성 검사, 기본 카테고리 접근 허용
- [ ] POST `/auth/login` — 자격증명 검증, Access + Refresh Token 발급
- [ ] POST `/auth/logout` — Refresh Token 무효화
- [ ] POST `/auth/refresh` — Access Token 재발급 (Refresh Token 유효성 검사)
- [ ] POST `/todos` — 유효성 검사, user_id·category_id 검증
- [ ] GET `/todos` — 필터(category_id, is_completed, schedule_status) 조합
- [ ] PUT `/todos/:id` — 본인 할일만 수정, 유효성 검사
- [ ] PATCH `/todos/:id/toggle` — completed_at 기록/초기화
- [ ] DELETE `/todos/:id` — 물리 삭제, 권한 검증

### UI/UX 검증
- [ ] 실시간 폼 유효성 검사 (클라이언트, 서버 검증 조합)
- [ ] 에러 메시지 인라인 표시 (필드 하단)
- [ ] 로딩 상태 표시 (스피너, 버튼 비활성화)
- [ ] 성공/실패 토스트 메시지
- [ ] 모달 확인 (삭제, 중요 작업)
- [ ] 반응형 레이아웃 (모바일 ≤ 1024px)

### 보안 검증
- [ ] 토큰 만료 후 자동 재발급 (인터셉터)
- [ ] 미인증 상태에서 보호 경로 리다이렉트
- [ ] 타인 할일 접근 시도 차단 (BR-03)
- [ ] SQL Injection 방지 (파라미터 바인딩)
- [ ] CORS 설정

### 데이터 무결성
- [ ] 날짜 유효성 (due_date ≥ start_date)
- [ ] 카테고리 필수 (BR-07)
- [ ] 할일 완료 상태와 completed_at 동기화
- [ ] overdue 필터 정의 (due_date < TODAY AND is_completed=false)
